# Design: Agent-First CLI Surface

## Overview

Add a CLI surface to Spectator and Director that is 1:1 in capabilities with
the MCP tools, designed for AI agents as the primary consumer. This is driven
by community consensus that CLIs are 10-32x cheaper and more reliable than MCP
for AI agent workflows (see Research section).

The CLI follows these agent-first principles:
- **JSON-in, JSON-out** — same field names, same response shapes as MCP
- **Stdout is data, stderr is logs** — clean separation for piping
- **One-shot execution** — connect, execute, print, exit
- **Meaningful exit codes** — 0 success, 1 failure, 2 usage error
- **Stdin support** — pipe JSON params from other tools

### Binary Naming Change

As part of this work, rename the `spectator-server` binary to `spectator`.
The binary serves dual duty (MCP server via `spectator serve`, CLI via
`spectator <tool> '<json>'`), matching the existing director pattern. The
crate name remains `spectator-server`.

### Command Surface

```bash
# Direct binary usage (agent-optimized):
spectator <tool> '<json-params>'
director <tool> '<json-params>'

# Via theatre wrapper (convenience):
theatre spectator <tool> '<json-params>'
theatre director <tool> '<json-params>'

# Stdin:
echo '{"detail":"summary","radius":50}' | spectator snapshot
cat params.json | director scene_create

# MCP mode (existing, renamed):
spectator serve     # stdio MCP server (was the default)
director serve      # stdio MCP server (unchanged)
```

### Statefulness

CLI mode is **one-shot**: each invocation connects to the Godot addon (for
Spectator) or launches Godot headless (for Director), executes the operation,
prints the result, and exits. No session state persists between calls.

This means stateful Spectator features (watches, delta baselines, spatial
index, config) operate within a single call only. Agents that need stateful
sessions should use MCP mode.

## Research: Why CLI over MCP

Community benchmarks (ScaleKit, 2026) on identical GitHub tasks:
- **Cost**: MCP required 4-32x more tokens due to schema injection
- **Reliability**: CLI achieved 100% success vs MCP's 72% (TCP timeouts)
- **Context**: MCP dumps all 43 tool schemas per call; CLI uses only what's needed

Agent-first CLI design patterns (DEV Community, InfoQ, 2026):
- JSON to stdout, everything else to stderr
- Noun-verb hierarchy for predictable command discovery
- Treat output as API contract — same shapes across CLI and MCP
- Support stdin for piping, `--quiet` for bare output
- Meaningful exit codes with structured error output

```rust
    let args: Vec<String> = std::env::args().collect();
```

### Unit 3: CLI One-Shot Executor (`cli.rs` in spectator-server)

**File**: `crates/spectator-server/src/cli.rs`

```rust
use anyhow::Result;
use serde_json::Value;
use std::io::Read;
use std::process::ExitCode;
use std::sync::Arc;
use tokio::sync::Mutex;

use crate::config;
use crate::tcp::{self, SessionState};

/// Known CLI tool names — must match MCP tool names exactly.
const TOOLS: &[&str] = &[
    "spatial_snapshot",
    "spatial_inspect",
    "scene_tree",
    "spatial_action",
    "spatial_query",
    "spatial_delta",
    "spatial_watch",
    "spatial_config",
    "clips",
];

/// Entry point for CLI one-shot mode.
///
/// 1. Parse params from argument or stdin
/// 2. Connect to Godot addon via TCP (one-shot)
/// 3. Dispatch to the appropriate handler
/// 4. Print JSON result to stdout
/// 5. Exit with appropriate code
pub async fn run(tool: &str, json_arg: Option<&str>) -> Result<()> {
    // Validate tool name
    if !TOOLS.contains(&tool) {
        let error = serde_json::json!({
            "error": "unknown_tool",
            "message": format!("Unknown tool: '{tool}'"),
            "available_tools": TOOLS,
        });
        eprintln!("{}", serde_json::to_string_pretty(&error)?);
        std::process::exit(2);
    }

    // Parse params: from argument, stdin, or empty object
    let params = parse_params(json_arg)?;

    // Resolve port
    let port = resolve_port();

    // Initialize minimal state
    let project_dir = std::env::var("SPECTATOR_PROJECT_DIR")
        .map(std::path::PathBuf::from)
        .unwrap_or_else(|_| std::env::current_dir().unwrap_or_default());
    let base_config = config::load_toml_config(&project_dir);
    let state = Arc::new(Mutex::new(SessionState {
        config: base_config,
        ..Default::default()
    }));

    // Connect to addon (one-shot, with timeout)
    let connected = tcp::connect_once(&state, port).await;
    if let Err(e) = connected {
        let error = serde_json::json!({
            "error": "connection_failed",
            "message": format!("Failed to connect to Godot addon on port {port}: {e}"),
            "hint": "Ensure the Godot project is running with the Spectator addon enabled.",
        });
        println!("{}", serde_json::to_string(&error)?);
        std::process::exit(1);
    }

    // Dispatch to handler
    let result = dispatch(tool, params, &state).await;

    match result {
        Ok(json_string) => {
            // MCP handlers return Result<String, McpError> where the
            // String is already serialized JSON. Print it directly.
            println!("{json_string}");
            std::process::exit(0);
        }
        Err(mcp_error) => {
            let error = serde_json::json!({
                "error": "tool_error",
                "code": mcp_error.code,
                "message": mcp_error.message,
            });
            println!("{}", serde_json::to_string(&error)?);
            std::process::exit(1);
        }
    }
}

/// Parse JSON params from CLI argument or stdin.
fn parse_params(json_arg: Option<&str>) -> Result<Value> {
    if let Some(json_str) = json_arg {
        // Explicit argument
        serde_json::from_str(json_str)
            .map_err(|e| anyhow::anyhow!("Invalid JSON params: {e}"))
    } else if atty::is_not(atty::Stream::Stdin) {
        // Stdin is piped — read from it
        let mut buf = String::new();
        std::io::stdin().read_to_string(&mut buf)?;
        let trimmed = buf.trim();
        if trimmed.is_empty() {
            Ok(Value::Object(Default::default()))
        } else {
            serde_json::from_str(trimmed)
                .map_err(|e| anyhow::anyhow!("Invalid JSON from stdin: {e}"))
        }
    } else {
        // No argument, no stdin — empty params
        Ok(Value::Object(Default::default()))
    }
}

/// Resolve TCP port from environment.
fn resolve_port() -> u16 {
    std::env::var("THEATRE_PORT")
        .or_else(|_| std::env::var("SPECTATOR_PORT"))
        .ok()
        .and_then(|v| v.parse().ok())
        .unwrap_or(9077)
}

/// Dispatch tool name to the appropriate MCP handler.
///
/// Reuses the exact same handler functions as MCP mode. The handlers
/// take typed params (deserialized from JSON) and return
/// Result<String, McpError>. We deserialize the raw JSON Value into
/// the typed params struct, then call the handler.
async fn dispatch(
    tool: &str,
    params: Value,
    state: &Arc<Mutex<SessionState>>,
) -> Result<String, rmcp::model::ErrorData> {
    use crate::mcp::*;
    use rmcp::model::ErrorData as McpError;

    match tool {
        "spatial_snapshot" => {
            let p: snapshot::SpatialSnapshotParams = deserialize_params(params)?;
            handle_snapshot(p, state).await
        }
        "spatial_inspect" => {
            let p: inspect::SpatialInspectParams = deserialize_params(params)?;
            handle_inspect(p, state).await
        }
        "scene_tree" => {
            let p: scene_tree::SceneTreeToolParams = deserialize_params(params)?;
            handle_scene_tree(p, state).await
        }
        "spatial_action" => {
            let p: action::SpatialActionParams = deserialize_params(params)?;
            handle_action(p, state).await
        }
        "spatial_query" => {
            let p: query::SpatialQueryParams = deserialize_params(params)?;
            handle_query(p, state).await
        }
        "spatial_delta" => {
            let p: delta::SpatialDeltaParams = deserialize_params(params)?;
            handle_delta(p, state).await
        }
        "spatial_watch" => {
            let p: watch::SpatialWatchParams = deserialize_params(params)?;
            handle_watch(p, state).await
        }
        "spatial_config" => {
            let p: config::SpatialConfigParams = deserialize_params(params)?;
            config::handle_spatial_config(p, state).await
        }
        "clips" => {
            let p: clips::ClipsParams = deserialize_params(params)?;
            handle_clips(p, state).await
        }
        _ => Err(McpError::invalid_params(
            format!("Unknown tool: {tool}"),
            None,
        )),
    }
}

/// Deserialize raw JSON Value into a typed params struct.
fn deserialize_params<T: for<'de> serde::Deserialize<'de>>(
    value: Value,
) -> Result<T, rmcp::model::ErrorData> {
    serde_json::from_value(value).map_err(|e| {
        rmcp::model::ErrorData::invalid_params(
            format!("Invalid parameters: {e}"),
            None,
        )
    })
}
```

**Implementation Notes**:
- **Handler reuse**: The `dispatch` function calls the same handler logic
  as the MCP `#[tool]` methods. This requires extracting handler logic from
  `SpectatorServer` methods into free functions (or calling through a thin
  adapter). See Unit 4 for the refactoring approach.
- **atty crate**: Used to detect stdin piping. Add `atty = "1"` to deps.
  Alternatively use `std::io::IsTerminal` (stable since Rust 1.70) — prefer
  the stdlib version: `use std::io::IsTerminal; !std::io::stdin().is_terminal()`.
- **Exit codes**: 0 = success, 1 = tool/connection error, 2 = usage error
  (unknown tool, invalid JSON syntax).
- **Error JSON**: Errors are printed to **stdout** as JSON (not stderr) so
  agents can always parse stdout as the result. Stderr is reserved for logs.
- **No activity logging**: CLI mode skips `log_activity` calls — there's no
  persistent session to log to.

**Acceptance Criteria**:
- [ ] `spectator spatial_snapshot '{}'` returns JSON snapshot to stdout
- [ ] `echo '{}' | spectator spatial_snapshot` reads params from stdin
- [ ] `spectator spatial_snapshot` with no args and no stdin uses empty params
- [ ] Invalid JSON prints error to stdout, exits with code 2
- [ ] Unknown tool prints error to stdout, exits with code 2
- [ ] Connection failure prints structured error to stdout, exits with code 1
- [ ] Tool handler errors print structured error to stdout, exits with code 1
- [ ] All 9 Spectator tools are dispatched correctly
- [ ] Response JSON matches MCP response shape exactly

### Unit 4: Extract Handler Functions from `SpectatorServer`

**File**: `crates/spectator-server/src/mcp/mod.rs` (refactor)

Currently, MCP handlers are methods on `SpectatorServer` (called via the
`#[tool_router]` impl). The CLI needs to call the same logic without
constructing a `SpectatorServer` instance.

**Approach**: Extract the handler body into free `async fn`s that take
`(params, &Arc<Mutex<SessionState>>)`. The `#[tool]` methods become thin
wrappers that call these + `log_activity`.

```rust
// Before (in #[tool_router] impl SpectatorServer):
#[tool(description = "...")]
pub async fn spatial_snapshot(
    &self,
    Parameters(params): Parameters<SpatialSnapshotParams>,
) -> Result<String, McpError> {
    let result = snapshot::handle_spatial_snapshot(params, &self.state).await;
    self.log_activity("query", &summary, "spatial_snapshot").await;
    result
}

// After — handler is already a free function (most handlers already follow
// this pattern). Just ensure all handlers are pub and take
// (params, &Arc<Mutex<SessionState>>) → Result<String, McpError>.
```

**Files to check/refactor**:
- `mcp/mod.rs` — the `#[tool_router]` impl. Most handlers already delegate
  to free functions in submodules. Verify all 9 do.
- `mcp/snapshot.rs` — `handle_spatial_snapshot` may need to be created if
  the logic is currently inline in the `#[tool]` method.
- Same for `inspect`, `scene_tree`, `action`, `query`, `delta`, `watch`,
  `config`, `clips`.

**Implementation Notes**:
- The snapshot handler currently has significant logic inline in the `#[tool]`
  method (perspective building, budget resolution, response building). This
  needs to be extracted into `snapshot::handle_snapshot(params, state)`.
- The clips handler returns `CallToolResult` (not `String`) for some actions
  (screenshot_at returns image content). For CLI mode, screenshots should be
  base64-encoded in the JSON response or saved to a file. Design decision:
  encode as base64 in a `{"image_base64": "...", "mime_type": "image/png"}`
  wrapper.

**Acceptance Criteria**:
- [ ] All 9 tool handlers callable as free functions
- [ ] MCP `#[tool]` methods are thin wrappers (delegate + log_activity)
- [ ] No behavior change in MCP mode
- [ ] CLI mode produces identical JSON output as MCP mode

### Unit 5: One-Shot TCP Connection (`tcp.rs` addition)

**File**: `crates/spectator-server/src/tcp.rs`

```rust
/// Connect to the Godot addon once (no reconnection loop).
/// Used by CLI one-shot mode.
///
/// Connects, performs handshake, and returns. The connection stays
/// alive in the background task until the process exits.
pub async fn connect_once(
    state: &Arc<Mutex<SessionState>>,
    port: u16,
) -> Result<()> {
    let addr = format!("127.0.0.1:{port}");
    let timeout = tokio::time::Duration::from_secs(5);

    let stream = tokio::time::timeout(timeout, tokio::net::TcpStream::connect(&addr))
        .await
        .map_err(|_| anyhow::anyhow!("Connection timed out after 5s"))?
        .map_err(|e| anyhow::anyhow!("TCP connection failed: {e}"))?;

    // Perform handshake and spawn background reader task
    // (reuses existing handle_connection logic)
    let state_clone = state.clone();
    tokio::spawn(async move {
        if let Err(e) = handle_connection(stream, state_clone).await {
            tracing::warn!("Connection ended: {e}");
        }
    });

    // Wait briefly for handshake to complete
    let handshake_timeout = tokio::time::Duration::from_secs(3);
    let start = tokio::time::Instant::now();
    loop {
        {
            let s = state.lock().await;
            if s.connected {
                return Ok(());
            }
        }
        if start.elapsed() > handshake_timeout {
            return Err(anyhow::anyhow!(
                "Handshake timed out — addon did not respond within 3s"
            ));
        }
        tokio::time::sleep(tokio::time::Duration::from_millis(50)).await;
    }
}
```

**Implementation Notes**:
- Reuses existing `handle_connection` which does the handshake and spawns
  the message reader. We just don't wrap it in a reconnection loop.
- The `connected` field on `SessionState` (or equivalent) indicates the
  handshake completed. If it doesn't exist, add a `connected: bool` field.
- 5s connect timeout + 3s handshake timeout = 8s max wait. Reasonable for
  CLI usage.
- The connection stays alive for the duration of the process. Since CLI mode
  runs one tool and exits, this is fine.

**Acceptance Criteria**:
- [ ] `connect_once` succeeds when addon is listening on the port
- [ ] `connect_once` returns error within 5s when nothing is listening
- [ ] `connect_once` returns error within 3s when TCP connects but handshake fails
- [ ] After `connect_once`, `query_addon` calls work correctly
- [ ] No reconnection loop — single attempt only

```rust
/// CLI one-shot mode — improved from original.
async fn cli(operation: &str, json_arg: Option<&str>) -> Result<()> {
    // Parse params from argument or stdin
    let params: serde_json::Value = if let Some(json_str) = json_arg {
        serde_json::from_str(json_str)
            .map_err(|e| {
                let error = serde_json::json!({
                    "error": "invalid_json",
                    "message": format!("Invalid JSON: {e}"),
                });
                println!("{}", serde_json::to_string(&error).unwrap_or_default());
                std::process::exit(2);
            })
            .unwrap()
    } else if !std::io::IsTerminal::is_terminal(&std::io::stdin()) {
        let mut buf = String::new();
        std::io::Read::read_to_string(&mut std::io::stdin(), &mut buf)?;
        let trimmed = buf.trim();
        if trimmed.is_empty() {
            serde_json::json!({})
        } else {
            serde_json::from_str(trimmed)
                .map_err(|e| anyhow::anyhow!("Invalid JSON from stdin: {e}"))?
        }
    } else {
        // No JSON provided — check if operation exists but params missing
        let error = serde_json::json!({
            "error": "missing_params",
            "message": format!("No JSON params provided. Usage: director {operation} '<json>'"),
        });
        println!("{}", serde_json::to_string(&error)?);
        std::process::exit(2);
    };

    let project_path = params
        .get("project_path")
        .and_then(|v| v.as_str())
        .ok_or_else(|| {
            let error = serde_json::json!({
                "error": "missing_project_path",
                "message": "params must include \"project_path\"",
            });
            println!("{}", serde_json::to_string(&error).unwrap_or_default());
            std::process::exit(2);
        })
        .unwrap();

    let godot = resolve_godot_bin().map_err(|e| {
        let error = serde_json::json!({
            "error": "godot_not_found",
            "message": format!("{e}"),
            "hint": "Ensure Godot is installed and in your PATH.",
        });
        println!("{}", serde_json::to_string(&error).unwrap_or_default());
        std::process::exit(1);
    }).unwrap();

    let project = std::path::Path::new(project_path);
    validate_project_path(project).map_err(|e| {
        let error = serde_json::json!({
            "error": "invalid_project",
            "message": format!("{e}"),
        });
        println!("{}", serde_json::to_string(&error).unwrap_or_default());
        std::process::exit(1);
    }).unwrap();

    let backend = Backend::new();
    let result = backend
        .run_operation(&godot, project, operation, &params)
        .await;

    backend.shutdown().await;

    match result {
        Ok(data) => {
            println!("{}", serde_json::to_string(&data)?);
            std::process::exit(0);
        }
        Err(e) => {
            let error = serde_json::json!({
                "error": "operation_failed",
                "message": format!("{e}"),
            });
            println!("{}", serde_json::to_string(&error)?);
            std::process::exit(1);
        }
    }
}
```

**Implementation Notes**:
- Director's existing CLI already works — this unit polishes it with:
  - Stdin support for piped JSON
  - Structured JSON error output (currently uses `anyhow` errors as text)
  - `--help` with categorized tool listing
  - `--version` with JSON output
  - Consistent exit codes (0/1/2)
- The `run_operation` call is unchanged — it already handles the full
  operation lifecycle (find Godot, launch headless, execute, return JSON).
- `serde_json::to_string` (not `to_string_pretty`) for CLI output — agents
  prefer compact JSON to minimize token usage. Humans can pipe through `jq`.

```rust
use anyhow::{Context, Result};
use std::process::{Command, Stdio};

use crate::paths;

#[derive(clap::Args)]
pub struct AgentArgs {
    /// Tool name (e.g., spatial_snapshot, scene_create)
    pub tool: String,

    /// JSON parameters (or pipe via stdin)
    pub params: Option<String>,
}

/// Dispatch `theatre spectator <tool> [params]` to the spectator binary.
pub fn run_spectator(args: AgentArgs) -> Result<()> {
    dispatch("spectator", &args.tool, args.params.as_deref())
}

/// Dispatch `theatre director <tool> [params]` to the director binary.
pub fn run_director(args: AgentArgs) -> Result<()> {
    dispatch("director", &args.tool, args.params.as_deref())
}

fn dispatch(binary: &str, tool: &str, params: Option<&str>) -> Result<()> {
    let bin_dir = paths::resolve_bin_dir()?;
    let bin_path = bin_dir.join(binary);

    if !bin_path.exists() {
        let error = serde_json::json!({
            "error": "binary_not_found",
            "message": format!("{binary} not found at {}", bin_path.display()),
            "hint": "Run `theatre install` first.",
        });
        println!("{}", serde_json::to_string(&error)?);
        std::process::exit(1);
    }

    let mut cmd = Command::new(&bin_path);
    cmd.arg(tool);

    if let Some(p) = params {
        cmd.arg(p);
    }

    // Inherit stderr for logging, inherit stdin for piping
    cmd.stderr(Stdio::inherit());
    cmd.stdin(Stdio::inherit());
    cmd.stdout(Stdio::inherit());

    let status = cmd.status().with_context(|| {
        format!("Failed to execute {}", bin_path.display())
    })?;

    std::process::exit(status.code().unwrap_or(1));
}
```

**Implementation Notes**:
- `theatre spectator snapshot '{}'` is exactly equivalent to
  `spectator snapshot '{}'`. The theatre wrapper just resolves the binary
  path from the install directory.
- Stdin, stdout, stderr are all inherited — the subprocess handles
  everything. The theatre binary is a transparent proxy.
- No tokio dependency needed — `std::process::Command` is synchronous.
- Binary lookup uses `resolve_bin_dir()` from paths.rs (same as install).

### Unit 8: Structured Error Contract

All CLI error output follows a consistent JSON envelope:

```rust
/// Standard CLI error response. Printed to stdout as JSON.
/// (Defined conceptually — each binary implements this inline,
/// no shared crate needed.)
///
/// Fields:
///   error:   string — error type identifier (snake_case)
///   message: string — human-readable description
///   hint:    string — optional actionable suggestion
///   code:    i32    — optional MCP error code (for tool errors)
///
/// Exit codes:
///   0 — success (stdout contains tool result JSON)
///   1 — runtime error (connection, operation, godot failure)
///   2 — usage error (unknown tool, invalid JSON, missing params)
```

Error type identifiers (the `error` field):

| Error Type | Exit Code | When |
|---|---|---|
| `unknown_tool` | 2 | Tool name not recognized |
| `invalid_json` | 2 | JSON parse failure |
| `missing_params` | 2 | Required params missing |
| `missing_project_path` | 2 | Director: no project_path |
| `connection_failed` | 1 | Spectator: can't reach addon |
| `godot_not_found` | 1 | Director: Godot not in PATH |
| `invalid_project` | 1 | project.godot not found |
| `operation_failed` | 1 | Tool handler returned error |
| `tool_error` | 1 | MCP handler error |
| `binary_not_found` | 1 | theatre wrapper: binary missing |

**Implementation Notes**:
- Errors go to **stdout** (not stderr) so agents always parse stdout.
  This is deliberate — agents read stdout, humans read stderr logs.
- The `hint` field provides actionable recovery steps when available.
- The `code` field is only present for tool_error (carries the MCP error
  code for debugging).

**Acceptance Criteria**:
- [ ] All error responses are valid JSON on stdout
- [ ] All error responses have `error` and `message` fields
- [ ] Exit codes are consistent across spectator, director, and theatre
- [ ] No raw text error messages — always structured JSON

### New dependencies

| Crate | Where | Why |
|---|---|---|
| (none) | spectator-server | Uses `std::io::IsTerminal` (stdlib) for stdin detection |

### Existing dependencies leveraged

| Dep | Where | Already in |
|---|---|---|
| `serde_json` | Error JSON construction | All crates |
| `tokio` | One-shot TCP, CLI async | spectator-server |
| `anyhow` | Error propagation | All crates |
| `clap` | Theatre wrapper args | theatre-cli |

No new external dependencies required. The design deliberately avoids adding
`atty` by using `std::io::IsTerminal` which is stable since Rust 1.70.

## Out of Scope

- **Persistent sessions**: CLI is one-shot only. Agents needing stateful
  sessions (watches, delta tracking) should use MCP mode.
- **Flag-based params**: No `--detail summary --radius 50` flags. JSON-only
  input matches MCP exactly and avoids maintaining 235+ flag definitions.
- **Output formatting**: No `--pretty` or `--compact` flags. Output is always
  compact JSON. Pipe through `jq` for human-readable formatting.
- **Shell completions**: Could be added later but not in initial scope.
- **Batch/pipe mode**: No JSONL batch processing. Each CLI call is one tool
  invocation. Director's `batch` tool handles multi-op batching.
